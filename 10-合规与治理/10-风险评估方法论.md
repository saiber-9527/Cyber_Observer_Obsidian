# 风险评估方法论

> 安全不是"花多少钱"的问题，是"把有限的钱花在最重要的风险上"的问题。

---

## 风险评估的核心公式

```
风险 = 威胁 × 脆弱性 × 资产价值

更实用的说法：
风险 = 发生的可能性 × 发生后的影响
```

---

## 常见风险评估方法

### 1. 定量评估

```
用钱说话：
  数据泄露概率：5%/年
  泄露后损失：$1000万
  风险值：$50万/年

  如果花 $40万/年 可以把这个风险降到 $5万/年
  → ROI = ($45万 - $40万) / $40万 = 12.5%
```

**优点：** 数字直观，可以算 ROI
**缺点：** 概率和损失很难精确估算

### 2. 定性评估

```
用等级说话：

可能性/影响 | 低      中      高
低          | 低风险  中风险  中风险
中          | 中风险  中风险  高风险
高          | 中风险  高风险  高风险
```

**优点：** 操作简单，不需要精确数据
**缺点：** 主观性强，不同人对等级的理解不同

### 3. 半定量评估

```
综合两者的做法：
可能性评分：1-5（低→高）
影响评分：   1-5（低→高）
风险评分：   可能性 × 影响 = 1-25

1-5:  低风险（可接受，定期追踪）
6-12: 中风险（需要控制措施，纳入计划）
13-25: 高风险（立即处理）
```

## 风险评估流程（ISO 27005）

```
1. 确定评估范围（哪些业务、系统、数据）
      │
2. 风险识别
      ├── 识别资产（有什么值得保护？）
      ├── 识别威胁（谁可能来攻击？）
      ├── 识别现有控制（已经在用什么措施？）
      └── 识别脆弱性（哪里还有漏洞？）
      │
3. 风险分析（评估可能性和影响）
      │
4. 风险评价（与风险接受标准比较）
      │
5. 风险处置
      ├── 降低（加控制措施）
      ├── 接受（啥也不做，接受风险）
      ├── 转移（买保险、外包）
      └── 规避（不做有风险的业务）
      │
6. 风险监控与回顾
```

---

## 风险处置决策

```
风险评分 25 分（高风险）→ 必须处理
  └── 降低：立刻加控制措施
  └── 规避：停掉这个业务
  └── 转移：买保险（但别依赖保险）

风险评分 12 分（中风险）→ 需要计划处理
  └── 降低：纳入安全改进计划
  └── 接受：如果控制成本 > 风险损失

风险评分 3 分（低风险）→ 可接受
  └── 接受：定期监控，不再恶化就行
```

## 风险场景示例

```yaml
场景：电商网站的客户数据泄露

资产：客户数据库（姓名、电话、地址、交易记录）
威胁：外部黑客攻击 / 内部员工泄密
脆弱性：数据库公网可访问、权限未最小化、审计日志不全

可能性：中（之前有类似案例，常见攻击手法）
影响：高（客户数据泄露 = 法律处罚 + 声誉损失 + 客户流失）
风险等级：高风险

处置方案：
  1. 立即：数据库改为内网访问（低）
  2. 1周内：创建应用专用账号，收回 root 权限（中）
  3. 1月内：启用数据库审计日志（中）
  4. 持续：每季度做一次数据访问权限审查（低）
```

## ⚠️ 常见错误

1. **风险登记册做完了就吃灰** — 风险是动态的，需要定期更新
2. **只关注外部威胁不管内部** — 内部威胁和外部威胁同样致命
3. **风险接受变成了"不作为"** — 接受风险不代表不监控，需要设定"如果恶化就处理"的条件
4. **低估了合规风险** — 法律法规的罚款也是一种重要的风险
5. **忽略供应链风险** — 供应商的安全漏洞也是你的风险
