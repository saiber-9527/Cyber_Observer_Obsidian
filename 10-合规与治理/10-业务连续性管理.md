# 业务连续性管理（BCM）

> 出了事能不能继续做生意——比安全更有发言权的业务维度。

---

## 什么是对业务连续性

```
安全关注：怎么防止出事
连续性关注：出了事怎么办、还能不能做生意
```

---

## 核心概念

### BCP vs DRP

```
BCP（业务连续性计划）— 业务层面，关注"怎么继续做生意"
  └── 应急操作流程（手工替代、场地切换、人员调配）

DRP（灾难恢复计划）— IT层面，关注"怎么恢复系统"
  └── 技术恢复流程（系统切换、数据恢复、网络重建）
```

### 关键指标

```
RTO（恢复时间目标）：从灾难到恢复的时间（"多久能好"）
RPO（恢复点目标）：可接受的数据丢失量（"丢多少数据"）

例如：
  核心银行系统：RTO < 1小时，RPO < 5分钟
  内部知识库系统：RTO < 4小时，RPO < 24小时
  归档日志：RTO < 72小时，RPO < 7天
```

---

## BCM 建设流程

```
1. 业务影响分析（BIA）
      └── 确定关键业务、关键流程、RTO/RPO
      │
2. 风险评估
      └── 分析可能导致业务中断的事件
      │
3. 制定策略
      └── 每个关键流程制定恢复方案
      │
4. 编写计划
      └── BCP 文档、DRP 文档、应急手册
      │
5. 培训与演练
      └── 桌面推演、技术演练、全面演习
      │
6. 维护与改进
      └── 定期评审、根据演练结果优化
```

## 业务影响分析（BIA）

### 业务分类矩阵

| 优先级 | 允许中断时间 | 影响 | 示例 |
|--------|-------------|------|------|
| **关键（Critical）** | < 1 小时 | 收入损失、法律处罚、客户流失 | 支付系统、客服中心 |
| **重要（Important）** | < 4 小时 | 运营效率下降 | 内部审批系统 |
| **辅助（Support）** | < 24 小时 | 轻度不便 | 内部文档系统 |
| **非关键（Non-critical）** | > 72 小时 | 无严重影响 | 历史数据归档 |

## 恢复策略

| 场景 | 策略 | 说明 |
|------|------|------|
| 服务器故障 | 主备切换 | Active-Passive 或 Active-Active |
| 数据中心故障 | 切换到灾备中心 | 同城/异地容灾 |
| 应用故障 | 回滚到上一个版本 | 版本控制 + 蓝绿部署 |
| 人员不可用 | 备份人员上岗 | 关键岗位至少两人 |
| 供应商中断 | 切换到备用供应商 | 关键供应商至少备选一家 |
| 网络中断 | 切换到备用链路 | 双 ISP 或 4G 备份 |

## BCP 文档模板

```yaml
# BCP 摘要

1. 适用范围：XX 电商平台
2. 触发条件：
   - 核心应用中断超过 5 分钟
   - 数据库不可用
   - 数据中心网络中断
3. 应急小组：
   - 总指挥：张三（CTO）
   - 技术组：李四、王五
   - 业务组：赵六
   - 通信组：钱七
   - 联系方：应急通讯录
4. 应急步骤：
   a. 触发应急 → 总指挥确认
   b. 30 分钟内切换至灾备环境
   c. 验证灾备环境服务正常
   d. 通知相关部门和客户
   e. 记录事故详情
5. 恢复确认：切换后持续监控 1 小时
6. 演练计划：每年 2 次桌面推演 + 1 次实战演练
```

## 演练类型

| 演练类型 | 说明 | 成本 | 效果 |
|---------|------|------|------|
| **桌面推演** | 一群人坐在一起讲"如果发生X，我们就做Y" | 低 | 中（发现流程问题） |
| **模拟演练** | 模拟部分恢复流程，不切换生产 | 中 | 较高 |
| **全面演练** | 实际切换至灾备系统运行一段时间 | 高 | 最高 |
| **突击演练** | 不通知的突发式演练 | 中 | 最高（最真实） |

## ⚠️ 常见问题

1. **BCP 写了但没练过** — 第一次演练发现"联系不上备份人员"，常见
2. **灾备环境和生产环境不一致** — 版本差了好几代，切换后启动不了
3. **备份了但没验证过恢复** — 备份文件可能损坏或不完整
4. **只关注 IT 恢复不关注业务** — 系统恢复了但没人知道怎么操作
5. **RTO 定得太激进** — 不现实的目标等于没有目标
