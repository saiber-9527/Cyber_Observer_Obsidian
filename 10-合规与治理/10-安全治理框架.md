# 安全治理框架

> 从组织层面系统性管理安全，而不是头痛医头。

---

## 什么是安全治理

```
治理（Governance）：
- 董事会/高管层定方向
- 确定安全策略和风险偏好
- 分配资源

管理（Management）：
- 执行层落地
- 实施控制措施
- 日常运营
```

**治理不做，安全就永远只是个技术问题而非业务问题。**

---

## 常见治理框架

### NIST 网络安全框架（CSF）

| 功能 | 核心活动 |
|------|---------|
| **Identify（识别）** | 资产管理、风险评估、治理 |
| **Protect（保护）** | 访问控制、数据安全、培训 |
| **Detect（检测）** | 异常事件、持续监控 |
| **Respond（响应）** | 响应计划、分析、缓解 |
| **Recover（恢复）** | 恢复计划、改进 |

### ISO 27001
- 基于 PDCA 的 ISMS 框架（见 [[10-ISO27001]]）

### COBIT
- IT 治理框架
- 关注 IT 与业务目标的一致性

## 风险管理

```
识别风险 → 分析风险 → 评价风险 → 处置风险 → 监控与回顾
                                              │
                                              └→ 循环
```

### 风险处置策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **降低** | 实施控制措施降低风险 | 多数场景 |
| **转移** | 买保险、外包 | 低概率高影响 |
| **规避** | 不做有风险的业务 | 风险太高无法接受 |
| **接受** | 不处理，接受剩余风险 | 低风险或成本太高 |

## 安全指标（KPI 示例）

| 维度 | 指标 |
|------|------|
| **预防性** | 漏洞修复率、补丁更新周期、员工安全意识测试通过率 |
| **检测性** | MTTD（平均发现时间）、告警降噪率 |
| **响应性** | MTTR（平均响应时间）、事件关闭率 |
| **合规性** | 等保测评通过率、审计发现数 |

## 组织架构建议

```
董事会
  └── 信息安全委员会（跨部门）
        ├── CISO
        │   ├── 安全运营团队
        │   ├── 安全架构团队
        │   └── 合规审计团队
        ├── 法务部（数据合规）
        └── IT 部门（技术实施）
```

---

## ⚠️ 关键要点

1. **安全治理的最终责任在业务领导** — 不能把安全责任全甩给 CISO
2. **安全策略要跟业务节奏走** — 太严业务跑不动，太松等于没有
3. **安全投入要能看到 ROI** — 避免的安全事故就是最大的收益
4. **安全文化建设比制度建设更重要** — 每个人都是安全的责任人
