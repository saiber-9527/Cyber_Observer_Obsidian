# 模型供应链安全

> 你用的模型/数据集/框架，来路都安全吗？

---

## 什么是模型供应链

```
数据采集 → 模型基座 → 微调/部署 → 应用集成 → 最终用户
   │          │           │          │           │
   数据来源    基座模型    微调框架    API/SDK    输出使用
   标注服务    第三方模型    插件      依赖库
```

**每一环都可能是攻击入口。**

---

## 供应链风险点

### 1. 数据集风险

| 风险 | 说明 | 后果 |
|------|------|------|
| 数据投毒 | 数据集中混入恶意样本 | 模型学到后门行为 |
| 版权侵权 | 数据集包含受版权保护的素材 | 法律风险 |
| 隐私泄露 | 数据集包含个人信息 | 违反数据保护法规 |
| 偏见污染 | 数据集分布有系统性偏差 | 模型输出偏见 |

**防御：** 数据溯源 → 数据清洗 → 数据标注审计 → 定期校验

### 2. 基座模型风险

```
下载 LLaMA 模型：
  │
  ├── 从官方渠道下载？ → ✅ 安全
  ├── 从第三方镜像站下载？→ ⚠️ 可能被篡改！
  └── 从网盘下载压缩包？  → ❌ 极度危险
  
  案例：2024 年发现伪装成 LLaMA 的恶意模型，
        在 Hugging Face 上被下载了数千次。
        模型解压后运行会植入木马。
```

**防御：**
- 只从官方渠道下载（Hugging Face 官方、GitHub Releases）
- 验证 checksum（官方发布的 SHA-256）
- 模型文件做签名验证

### 3. 微调框架/工具链风险

```bash
pip install some-ai-package  # 这个包安全吗？
npm install ai-plugin     # 这个插件安全吗？
```

- PyPI/NPM 可能有同名恶意包（Typosquatting）
- 第三方 Plugin 可能窃取对话数据
- 依赖库的依赖也存在漏洞（供应链嵌套）

### 4. 部署环境风险

- 模型部署在公有云 → 云厂商是否有数据访问权限？
- 使用 GPU 提供商 → 显存中的数据是否会被其他租户读取？
- API 网关 → 敏感 prompt 是否被中间件记录？

---

## 供应链安全最佳实践

```
采购环节：
  ├── 供应商安全评估（ISO 27001 / SOC2）
  ├── 模型来源验证（checksum + 签名）
  └── 数据来源审计（是否含敏感数据）

开发环节：
  ├── 依赖扫描（SCA 工具：Snyk / Trivy）
  ├── 容器镜像扫描（Trivy / Clair）
  └── 代码审计

部署环节：
  ├── 模型容器化（隔离运行环境）
  ├── 访问控制（模型 API + 数据 API）
  └── 日志审计（谁在使用模型、用什么数据）

运维环节：
  ├── 持续监控模型行为变化（是否有异常输出）
  ├── 定期重新评估供应链风险
  └── 厂商变更时重新做安全评估
```

## SBOM（软件物料清单）在 AI 领域

SBOM = Software Bill of Materials（软件物料清单）

```
传统 SBOM：列出所有代码依赖
AI SBOM 扩展：
  ├── 训练数据来源及版本
  ├── 基座模型及其版本号
  ├── 微调数据集信息
  ├── 使用的标注服务信息
  └── 所有 Python 依赖和版本
```

**目的：** 出问题时知道"哪个环节引入了风险"，快速定位和修复。
