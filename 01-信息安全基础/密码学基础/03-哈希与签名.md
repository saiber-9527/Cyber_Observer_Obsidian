# 哈希函数与数字签名

---

## 哈希函数（Hash Function）

### 特点
- **固定长度输出** — 输入任意长，输出固定长（如 SHA-256 输出 256 bit）
- **不可逆** — 从哈希值不可能还原出原始数据
- **碰撞抗性** — 极难找到两个不同输入产生同一哈希值
- **雪崩效应** — 输入改一位，输出天差地别

### 常见算法

| 算法 | 输出长度 | 状态 | 说明 |
|------|---------|------|------|
| **SHA-256** | 256 bit | ✅ 推荐 | 当前最广泛使用 |
| **SHA-3** | 可变 | ✅ 推荐 | 新一代标准 |
| **MD5** | 128 bit | ❌ 已攻破 | 碰撞攻击可行，不要用 |
| **SHA-1** | 160 bit | ❌ 不推荐 | 已被碰撞攻击 |
| **SM3** | 256 bit | ✅ 国密 | 中国标准 |

### 典型用途
- **密码存储**：存储 `hash(password + salt)`，不存明文
- **完整性校验**：下载文件后比对该文件的 SHA-256
- **数据指纹**：Git 提交 ID、区块链
- **消息验证码 HMAC**：带密钥的哈希，用于 API 签名

## 数字签名（Digital Signature）

### 工作流程

```
签名过程（签名者）：
  原文 → SHA-256 → 摘要 → RSA/ECC 私钥加密 → 签名

验证过程（验证者）：
  原文 → SHA-256 → 摘要A
  签名 → 公钥解密 → 摘要B
  比对摘要A = 摘要B ✅ 签名有效
```

### 核心价值
1. **完整性** — 原文被篡改后签名验证失败
2. **真实性** — 只有持有私钥的人才能签名
3. **不可否认性** — 签名者事后不能抵赖

### 典型场景
- 软件包签名（apt 签名、npm 包签名）
- 代码提交签名（Git 用 GPG/SSH 签名）
- HTTPS 证书签发机构（CA）
- 智能合约交易签名

---

## ⚠️ 陷阱

1. **MD5/SHA-1 仍然在旧系统中使用** — 确认一下自己的系统是否还依赖它们
2. **哈希不等于加密** — 哈希不可逆，加密可逆，别搞混
3. **不加盐的密码哈希** — 同样的密码产生同样的哈希值，彩虹表秒破
4. **签名前不看原文** — 别在不知道签了什么的情况下签名

## 速查表

| 需要什么 | 用什么 |
|---------|--------|
| 完整性校验 | SHA-256 哈希 |
| 防篡改+防抵赖 | 数字签名（SHA-256 + RSA/ECC） |
| 安全密码存储 | bcrypt / argon2 / scrypt |
| API 请求认证 | HMAC-SHA256 |
| 国密合规 | SM3（哈希）、SM2（签名） |
