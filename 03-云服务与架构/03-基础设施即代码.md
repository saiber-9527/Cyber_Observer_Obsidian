# 基础设施即代码（IaC）

> 用代码管理基础设施——可重复、可审计、可版本控制。

---

## 什么是 IaC

传统方式：
```
运维人员：手动在云控制台点击创建服务器
问题：谁改了什么东西？上次怎么配的来着？咋重复？
```

IaC 方式：
```hcl
# Terraform 代码定义基础设施
resource "alicloud_instance" "web" {
  instance_name   = "web-server"
  image_id        = "centos_7_9_x64_20G_alibase_2023"
  instance_type   = "ecs.g6.large"
  vswitch_id      = alicloud_vswitch.public.id
  security_groups = [alicloud_security_group.web.id]
  
  tags = {
    Name    = "web-server"
    Env     = "production"
    Managed = "terraform"
  }
}
```

**优势：** 代码→审查→执行→版本控制→可回滚

## 主流 IaC 工具对比

| 工具 | 语言 | 状态管理 | 适用场景 |
|------|------|---------|---------|
| **Terraform / OpenTofu** | HCL | 有状态（state file） | 多云，基础设施全生命周期 |
| **Pulumi** | Python/Go/TS | 有状态 | 开发者友好 |
| **CloudFormation** | YAML/JSON | AWS 原生 | 只用 AWS |
| **Ansible** | YAML | 无状态（幂等） | 配置管理 + 应用部署 |
| **Packer** | HCL | — | 创建机器镜像 |
| **CDK（AWS）** | Python/TS | 基于 CloudFormation | AWS 开发者 |

## Terraform 安全实践

### 1. 状态文件保护

```bash
# 状态文件包含基础设施的全部信息（包括可能的敏感字段）
# 绝对不要提交状态文件到 Git！

# 正确做法：使用远程状态存储（带加密）
terraform {
  backend "oss" {  # 阿里云 OSS
    bucket = "my-terraform-state"
    key    = "production/terraform.tfstate"
    region = "cn-shanghai"
    encrypt = true
  }
}
```

### 2. 敏感信息管理

```hcl
# 不要硬编码密码
resource "alicloud_rds_instance" "db" {
  # ❌ 错误方式
  # password = "MyP@ssw0rd!"
  
  # ✅ 正确方式：使用变量 + 环境变量或 Key Vault
}

# variables.tf
variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true  # 标记为敏感，不显示在输出中
}

# 运行时通过环境变量传入
# export TF_VAR_db_password="MySecureP@ss!"
```

### 3. 安全扫描（IaC 扫描）

```bash
# 在 CI/CD 中集成 IaC 安全扫描

# Checkov（最流行的 IaC 扫描工具）
checkov -d . --framework terraform

# tfsec（专注 Terraform 安全）
tfsec .

# Terrascan
terrascan scan -d .
```

**Checkov 发现的问题示例：**
```bash
Check: CKV_ALICLOUD_5: "Ensure OSS bucket has public access blocked"
FAILED for resource: alicloud_oss_bucket.my_bucket
  File: main.tf:10-15
  Guide: https://docs.bridgecrew.io/docs/...

Check: CKV_ALICLOUD_1: "Ensure RDS instance has SSL enabled"
PASSED for resource: alicloud_rds_instance.db
```

### 4. 最小权限原则

```hcl
# 给实例配置 IAM 角色，而不是在实例中放 AccessKey
resource "alicloud_ram_role" "ecs_role" {
  name = "ecs-app-role"
  document = <<EOF
{
  "Statement": [
    {
      "Action": "sts:AssumeRole",
      "Effect": "Allow",
      "Principal": {
        "Service": ["ecs.aliyuncs.com"]
      }
    }
  ]
}
EOF
}

# 为角色绑定最小权限策略
resource "alicloud_ram_role_policy_attachment" "oss_readonly" {
  role_name   = alicloud_ram_role.ecs_role.name
  policy_name = "AliyunOSSReadOnlyAccess"  # 只读，不是完全控制
}
```

## 安全 IaC 清单

```
□ 状态文件使用远程存储（加锁 + 加密）
□ 敏感信息使用变量 + 环境变量/Key Vault
□ IaC 代码纳入代码审查流程
□ CI/CD 中集成 IaC 安全扫描（Checkov/tfsec）
□ 使用 Terraform Plan 预览变更后再 Apply
□ 基础设施变更走审批流程
□ 定期审计基础设施状态与 IaC 代码的一致性（避免漂移）
□ 删除不再使用的资源（避免产生僵尸资源）
```

---

## ⚠️ 常见陷阱

1. **状态文件丢失 = 基础设施失控** — Terraform state 是命根子，必须备份
2. **手工修改云资源不更新 IaC 代码** — 产生"配置漂移"，下次 Apply 会回滚手工修改
3. **Terraform Plan 不仔细看就直接 Apply** — Apply 会把删掉代码的资源也删掉
4. **把生产环境和测试环境用一个 state 管理** — 应该分开不同的 state 文件
