# VPN 与零信任

> 从"先连进来再说"到"逐个验证才能访问"。

---

## VPN（虚拟私有网络）

### 工作原理
- 在公共网络上建立加密隧道
- 远程用户看起来就像"在内网"

### 常见协议
| 协议 | 特点 | 状态 |
|------|------|------|
| IPsec | 成熟稳定 | 仍广泛使用 |
| OpenVPN | 开源灵活 | ✅ 推荐 |
| WireGuard | 极简高性能 | ✅ 新项目首选 |
| SSTP | Windows 原生 | 平台绑定 |
| L2TP/IPsec | 老旧 | ❌ 不推荐 |

### VPN 的问题
1. **攻击面大** — 一旦连入，内网几乎所有资源暴露
2. **横向移动风险** — 感染一台设备后容易扩散
3. **性能瓶颈** — 所有流量经由 VPN 网关
4. **运维复杂** — 证书管理、客户端部署

## 零信任（Zero Trust）

### 核心原则

```
永不信任，始终验证。
最小权限，逐段隔离。
```

### NIST SP 800-207 零信任架构
1. **所有数据源和计算服务视为资源**
2. **无论网络位置，所有通信必须安全**
3. **逐会话授予资源访问权限**
4. **动态策略评估**（身份、设备、行为、上下文）
5. **持续监控并重新评估信任**

### 关键组件
- **SDP（软件定义边界）** — 隐藏服务，默认拒绝连接
- **ZTNA（零信任网络访问）** — Zscaler、Cloudflare Access
- **IAM + MFA** — 身份是信任的基石
- **EDR / XDR** — 持续监控设备状态

### VPN → ZTNA 对比

| 维度 | VPN | ZTNA |
|------|-----|------|
| 访问模型 | 先连网后授权 | 先认证后访问 |
| 暴露面 | 整个内网 | 逐个应用 |
| 横向移动 | 容易 | 难以实现 |
| 用户体验 | 需要客户端 | 浏览器即可 |

---

## 实践建议

- **新建系统优先考虑零信任架构**（Cloudflare Access、Zscaler、阿里云 SDP）
- **存量 VPN 逐步迁移**—不要一次性切换，先对非关键系统做试点
- **零信任不是一项产品** —是架构思想，需要身份、设备、网络多层面配合
