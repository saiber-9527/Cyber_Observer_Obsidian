# 向量数据库对比

> RAG 应用的核心组件——找到最相关的文档片段。

---

## 什么是向量数据库

- 存储和检索**向量**（浮点数数组）
- 支持**近似最近邻搜索**（ANN）
- 常见的 RAG 流程入口

---

## 主流方案对比

| 方案 | 部署方式 | 特点 | 适用场景 |
|------|---------|------|---------|
| **Chroma** | 轻量嵌入式 | 最易用，Python 原生 | 开发/小规模 |
| **FAISS** | 库（非数据库） | Meta 出品，性能极好 | 离线/准实时检索 |
| **Milvus** | 分布式服务 | 云原生，大规模生产 | 企业级 RAG |
| **Pinecone** | 云服务 | 全托管，零运维 | 无需自建 |
| **Qdrant** | 服务 | Rust 实现，高性能 | 中等规模 |
| **Weaviate** | 服务 + Graph | 可同时做向量+标量搜索 | 混合搜索 |
| **Elasticsearch** | 服务 | 8.0 后支持向量搜索 | 已有 ES 生态 |
| **PostgreSQL + pgvector** | 数据库插件 | 不用额外维护一个系统 | 已有 PG 的场景 |

---

## 选型建议

```
规模小（< 100万向量）：
  ├── 开发测试 → Chroma（最省事）
  ├── 已有 PostgreSQL → pgvector
  └── 离线场景 → FAISS

规模中等：
  ├── 需要全托管 → Pinecone / Qdrant Cloud
  └── 自建 → Qdrant / Milvus Standalone

大规模生产（> 1000万向量）：
  ├── 自建 → Milvus Cluster
  └── 云上 → Zilliz Cloud（Milvus 托管版）
```

## 关键指标

| 指标 | 说明 | 影响 |
|------|------|------|
| **向量维度** | 每个向量的浮点数数量 | 越高内存越大，搜索越慢 |
| **索引类型** | IVF、HNSW、DiskANN | 影响搜索速度和精度 |
| **Recall 召回率** | 返回的结果中有多少是真正最相似的 | 精度 vs 速度的 trade-off |
| **QPS** | 每秒查询数 | 决定了并发能力 |
| **CRUD** | 是否能快速增删改 | 数据更新频繁的场景重要 |

---

## 向量检索原理（HNSW 示例）

```
HNSW（分层可导航小世界图）：
  上层：粗粒度索引（快速缩小范围）
  下层：细粒度索引（精准搜索结果）
  
  搜索时：从上到下，逐层缩小范围
  速度：百万级向量搜索毫秒级返回
```

---

## ⚠️ 实践踩坑

1. **向量维度不是越大越好** — 1536 维的嵌入需要大内存，256-768 维通常够用
2. **批量导入考虑写入性能** — 一次性导入百万向量可能耗时很长
3. **数据量大记得做分区** — 按时间/类别分片，提升搜索效率
4. **监控内存使用** — 向量索引全部在内存中，内存决定能存多少数据
