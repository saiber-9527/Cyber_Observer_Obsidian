# 云原生安全

> K8s 和容器带来的新安全挑战——不是在传统安全上加壳，而是推翻了重来。

---

## 容器安全生命周期

```
Build ──→ Ship ──→ Run
 │         │        │
 镜像安全   仓库安全   运行时安全
 │         │        │
 ├ 基础镜像选择    ├ 镜像签名  ├ 容器逃逸检测
 ├ 镜像漏洞扫描    ├ 镜像仓库ACL├ 网络策略
 ├ Dockerfile安全  ├ 供应链验证 ├ Runtime 监控
 └ 最小化镜像      └ 镜像拉取限├ 资源限制
                                  └ Seccomp/AppArmor
```

## K8s 安全核心关注点

### 1. RBAC（基于角色的访问控制）
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: { namespace: default, name: pod-reader }
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```
⚠️ 注意：`cluster-admin` 权限不要随便绑定

### 2. Pod 安全策略（Pod Security Admission）
```yaml
# 禁止特权容器
securityContext:
  runAsNonRoot: true
  capabilities:
    drop: ["ALL"]
```

### 3. 网络策略（NetworkPolicy）
- 默认 Pod 之间全通——你需要显式隔离
- 使用 Calico / Cilium 做微隔离

### 4. 镜像扫描
- 所有容器镜像在部署前需要扫描
- Trivy、Clair、Anchore

### 5. Secret 管理
- K8s Secret 默认只是 base64 编码，不是加密！
- 用外部密钥管理（Vault、阿里云 KMS、AWS KMS）

## 常见攻击路径

```
外部 ──→ 未认证的 kube-apiserver 暴露
        → 容器漏洞应用代码
        → 供应链攻击（恶意镜像）

内部 ──→ 容器逃逸到宿主机
        → 横向移动到其他 Pod
        → 权限提升到集群管理员
        → 控制 etcd = 控制整个集群
```

---

## ⚠️ 陷阱

1. **ImagePullPolicy 为 Always** — 启动时每次从外部拉镜像，供应链攻击窗口
2. **privileged 容器** — 容器可以访问宿主机全部设备，等同于不限
3. **不设资源限制** — 一个容器的内存泄漏可能拖垮整个节点
