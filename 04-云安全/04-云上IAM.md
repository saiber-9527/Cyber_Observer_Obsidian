# 云上 IAM（身份与访问管理）

> 云安全的第一道门——管好人、权限和信任。

---

## IAM 核心组件

以阿里云为例，AWS 概念类似：

| 组件 | 说明 | 安全要点 |
|------|------|---------|
| **主账号（根用户）** | 账号拥有者，最高权限 | 日常不要使用！开启 MFA |
| **RAM 用户** | IAM 用户，可以分配权限 | 遵循最小权限原则 |
| **RAM 角色** | 给 AWS 服务/信任实体用的身份 | 优先用角色替代密钥 |
| **权限策略** | 定义允许/拒绝什么操作 | 以 Action + Resource + Condition |
| **策略引用** | 用户/角色绑定策略 | 定期审计 |

---

## 最小权限实践

```json
{
  "Version": "1",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["oss:GetObject", "oss:PutObject"],
      "Resource": "acs:oss:*:xxxxx:my-bucket/*"
    }
  ]
}
```

**给策略的思路：**
1. 先不配任何权限
2. 按需逐步添加最小范围
3. 定期使用 IAM Access Advisor（访问分析器）审查未使用的权限

---

## 常见陷阱

| 陷阱 | 风险 |
|------|------|
| 根账号日常使用 | 密钥泄露 = 整个账户沦陷 |
| 长期 AccessKey 硬编码 | 代码仓库泄露就完了 |
| 权限过大 | 违反最小权限 |
| 用户清理不及时 | 离职员工仍有权限 |
| 信任策略太宽 | 让非本账号的实体也能访问 |

---

## 最佳实践

1. **根用户开 MFA，锁起来** — 只在关键操作时才用它
2. **优先用 RAM 角色而非用户** — 临时凭证，自动轮转
3. **AccessKey 定期轮转** — 90 天换一次
4. **不使用长期 AccessKey**时用 STS（临时令牌）
5. **权限不直接绑用户** — 通过用户组/角色管理
