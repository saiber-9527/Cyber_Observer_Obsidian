# 云上日志与监控

> 看不见就不安全。

---

## 为什么要云上日志

传统环境的日志：翻服务器文件。
云上日志的挑战：
- 资源数量多（几十台 ECS 是常事）
- 日志分散（不同服务不同日志）
- 动态伸缩（资源随时增减，日志源变化）

---

## 核心日志类型

| 类别 | 阿里云 | AWS | 说明 |
|------|--------|-----|------|
| **操作审计** | ActionTrail | CloudTrail | 谁在什么时候调了什么 API |
| **访问日志** | OSS 访问日志 | S3 Access Logs | 谁访问了对象存储 |
| **网络日志** | 流日志（VPC Flow） | VPC Flow Logs | 哪些 IP 在访问你的哪些资源 |
| **安全告警** | 云安全中心 | GuardDuty | 检测到的安全事件 |
| **主机日志** | 日志服务（SLS） | CloudWatch | 应用的 stdout、错误日志 |

---

## 监控架构建议

```
日志源
 ├── 操作审计日志 → 发送到 日志服务（SLS）
 ├── VPC 流日志   → 发送到 日志服务（SLS）
 ├── 安全告警     → 发送到 云安全中心
 ├── 应用日志     → 通过 Logtail 采集到 SLS
 └── 容器日志     → 通过 Fluentd 采集到 SLS
        │
        ▼
日志服务（SLS / CloudWatch）
 ├── 实时告警（邮件/短信/钉钉）
 ├── 可视化仪表盘
 └── 长期归档（OSS / S3 Glacier）
```

---

## 告警策略设置原则

1. **少而精** — 告警太多 = 没人看，关键告警会被淹没
2. **分优先级** — P0（业务中断）→ P1（安全事件）→ P2（异常但可观察）
3. **带上下文** — 告警消息里写明：发生了什么、影响什么资源、建议操作
4. **避免告警风暴** — 设置聚合窗口（告警合并后发一条）
5. **定期验证** — 你的告警通道真的能收到消息吗？

---

## 最佳实践

- **日志不可篡改** — 写入日志服务后只允许追加，不允许修改/删除
- **日志集中化** — 不要让日志分散在每个资源上
- **最小日志保留周期** — 至少 180 天（等保要求），关键日志 1 年以上
- **成本控制** — 日志存储费用不低，区分热存储（30 天）和冷归档（长期）
