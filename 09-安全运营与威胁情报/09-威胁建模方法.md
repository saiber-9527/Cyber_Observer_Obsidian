# 威胁建模方法

> 在攻击发生之前，先想想"敌人可能会怎么打我"。

---

## 什么是威胁建模

```
威胁建模 = 结构化地识别和分析潜在威胁
            │
            ├── 不是等出了事再补救
            ├── 不是凭直觉"感觉这里不安全"
            └── 是用系统的方法找出最需要防护的地方
```

---

## 主流威胁建模方法

### 1. STRIDE（微软）

STRIDE 按威胁类型分类，每种对应需要保护的属性：

| 首字母 | 威胁 | 对应属性 | 典型问题 |
|-------|------|---------|---------|
| **S** | 身份欺骗 (Spoofing) | 认证 | 攻击者冒充其他用户 |
| **T** | 数据篡改 (Tampering) | 完整性 | 中间人修改数据 |
| **R** | 抵赖 (Repudiation) | 不可否认性 | 用户否认做过某操作 |
| **I** | 信息泄露 (Information Disclosure) | 机密性 | 敏感数据被窃听 |
| **D** | 拒绝服务 (Denial of Service) | 可用性 | 系统被 DDoS |
| **E** | 权限提升 (Elevation of Privilege) | 授权 | 普通用户变成了管理员 |

**使用方法：** 对系统的每个组件逐一过一遍 STRIDE，看看存在哪些威胁。

```yaml
示例：对"登录模块"做 STRIDE 分析

Spoofing:     攻击者能否伪造其他用户的令牌？
Tampering:    登录请求能否被中间人篡改？
Repudiation:  用户能否抵赖"我没登录过"？
Information:  密码是否明文传输/存储？
DoS:          登录接口是否会被频繁调用导致拥堵？
Elevation:    普通用户能否通过登录漏洞变成管理员？
```

### 2. PASTA（攻击模拟与威胁分析过程）

```
7 步骤系统化分析：

Step 1: 定义业务目标和技术范围
Step 2: 分解系统（数据流图、信任边界）
Step 3: 识别和分类资产
Step 4: 分析威胁（基于 ATT&CK 等框架）
Step 5: 分析漏洞
Step 6: 枚举攻击路径
Step 7: 评估风险和优先级

特点：业务导向，输出"需要修复什么"的优先级清单
```

### 3. Attack Trees（攻击树）

```
攻击目标：窃取数据库中的用户数据
  │
  ├── OR ── 外部攻击
  │        ├── SQL 注入 → Web 应用漏洞
  │        ├── 钓鱼管理员 → 获得凭证
  │        └── 零日漏洞 → 未公开 CVE
  │
  └── OR ── 内部攻击
           ├── 离职员工泄密 → 权限未回收
           └── 运维人员窃取 → 特权滥用
```

**特点：** 以攻击者视角倒推路径，直观展示攻击链

### 4. LINDDUN（隐私威胁建模）

```
专注隐私泄露的威胁建模方法
  ├── Linkability（可关联性）
  ├── Identifiability（可识别性）
  ├── Non-repudiation（不可否认性）— 隐私角度
  ├── Detectability（可检测性）
  ├── Disclosure of information（信息披露）
  ├── Unawareness（用户不知情）
  └── Non-compliance（不合规）

适用于需要做隐私合规（GDPR/个保法）的系统
```

## 威胁建模流程

```
1. 界定范围 → 分析哪些系统/数据/用户需要建模

2. 分解系统 → 画数据流图（DFD），标出：
   - 外部实体（用户、第三方系统）
   - 处理过程（服务、函数）
   - 数据存储（数据库、缓存）
   - 数据流（请求/响应）
   - 信任边界（内网外网的分界）

3. 识别威胁 → 用 STRIDE/PASTA 逐项检查

4. 评估风险 → 按影响×概率给每个威胁打分

5. 确定对策 → 对高风险威胁提出缓解措施

6. 记录和跟踪 → 威胁模型需要维护（系统变了就重做）
```

用户（外部实体）

## ⚠️ 常见误区

1. **做一次就不再管了** — 系统在不断变，威胁模型需要定期更新
2. **做太细陷入细节** — 先把握大方向，把高风险威胁找出来，不要追求"覆盖所有可能"
3. **只做技术威胁不考虑业务威胁** — 业务层面（如：用户恶意退款）也是威胁
4. **威胁模型做了但不落地** — 发现的问题要转化成具体的工单去修复
