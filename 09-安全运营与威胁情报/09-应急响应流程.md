# 应急响应流程

> 安全事故发生时，冷静的流程比个人能力更重要。

---

## 响应六阶段（NIST SP 800-61）

```
准备 → 检测与分析 → 遏制 → 根除 → 恢复 → 事后复盘
```

### 1. 准备（Preparation）
- 应急预案是否齐全？
- 应急工具是否就绪？（取证工具、备用机器）
- 人员联系方式是否最新？

### 2. 检测与分析（Detection & Analysis）
- 发生了什么？影响范围？
- 攻击入口在哪里？
- 攻击者做了什么？

### 3. 遏制（Containment）
- **短期遏制**：阻断攻击 IP、隔离主机、下线服务
- **长期遏制**：系统打补丁、修改访问策略

### 4. 根除（Eradication）
- 删除恶意文件
- 移除后门
- 修复漏洞

### 5. 恢复（Recovery）
- 从备份恢复数据
- 逐步恢复业务服务
- 持续监控确保攻击者没有回来

### 6. 事后复盘（Post-mortem）
- 时间线还原
- 根因分析
- 改进措施
- 经验沉淀

---

## 应急响应黄金时间

```
发现 ── 1小时 ── 遏制 ── 4小时 ── 根除 ── 24h ── 恢复 ── 72h ── 复盘
 │                  │                   │                    │
 发现越早越好       遏制越快损失越小      根除要彻底            经验要沉淀
```

---

## 响应 SOP 示例

```
事件发现 → 事件登记 → Triage 研判 → 定级 → 启动响应

P0（严重）：核心业务受影响，数据可能泄露
  → 立即建群拉人
  → 启动遏制措施
  → 汇报管理层

P1（高危）：确定的安全事件，非核心业务
  → 1 小时内处置
  → 遏制后分析根因

P2（中危）：可疑活动，尚未确认事件
  → 4 小时内确认

P3（低危）：参考信息
  → 正常排期处理
```

---

## ⚠️ 常见错误

1. **应急处置"烧 CPU"不"烧流程"** — 临场发挥不如按流程走
2. **草率恢复业务** — 根因没找到就恢复，攻击者还在里面
3. **证据不保存** — 复盘需要完整的时间线和日志
4. **忘记通知** — 合规（等保/GDPR）要求在特定时限内通知监管机构
