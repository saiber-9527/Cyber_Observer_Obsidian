# 日志分析实战

> 安全运营的本质工作就是"看日志"——但要知道看什么、怎么看。

---

## 哪些日志需要看

| 日志类型 | 数据量 | 保留周期 | 关键关注点 |
|---------|--------|---------|-----------|
| **Web 访问日志** | 大 | 30-90天 | 异常 URI、异常 User-Agent、扫描特征 |
| **数据库审计日志** | 中 | 180天 | 异常 SQL（drop/delete/大量查询） |
| **主机安全日志** | 中 | 180天 | 可疑进程、文件变动、登录失败 |
| **网络流量日志** | 极大 | 30天 | 异常外连、大流量 |
| **应用业务日志** | 大 | 30-90天 | 业务异常（刷单、越权） |
| **云操作审计日志** | 中 | 180天 | 配置变更、权限修改 |
| **防火墙/安全组日志** | 大 | 90天 | 被阻断的访问尝试 |

---

## 日志分析三问

```
每看到一条日志，问自己三个问题：

1. 这是正常行为吗？（建立基线）
    ── 一个服务器的 SSH 登录，平时每天 5 次，今天突然 500 次

2. 这个行为是"谁"做的？
    ── 是 admin 在例行运维，还是未知 IP 在爆破

3. 这个行为导致了"什么后果"？
    ── 登录成功了吗？有后续操作吗？数据被读取了吗？
```

## 日志分析命令

### 标准 Linux 日志分析

```bash
# 登录失败分析
# 查看 login 失败记录
grep "Failed password" /var/log/secure | awk '{print $11}' | sort | uniq -c | sort -rn | head -10

# 查看 SSH 登录成功记录
grep "Accepted password" /var/log/secure | awk '{print $9}' | sort | uniq -c | sort -rn

# 最近的登录记录
last -10

# 查看 sudo 使用记录
grep "sudo" /var/log/secure

# crontab 变更审计
grep "CRON" /var/log/cron

# 异常进程分析
ps aux --sort=-%cpu | head -10
ps aux --sort=-%mem | head -10

# 查看非标准端口的监听
ss -tlnp | grep -vE ':22|:80|:443|:3306|:6379'

# 文件修改审计
find /etc -mtime -1 -type f
```

### 使用 /var/log 下的关键文件

```bash
# 检查这些文件有没有异常
/var/log/secure      # 认证日志（SSH、sudo）
/var/log/messages    # 系统通用日志
/var/log/maillog     # 邮件日志
/var/log/cron        # 定时任务日志
/var/log/dmesg       # 内核日志（硬件问题/异常）
/var/log/httpd/      # Apache/Nginx 日志
/var/log/audit       # auditd 审计日志
```

## 告警规则编写示例

### 阿里云 SLS（日志服务）告警

```sql
-- 检测 SSH 爆破
SELECT
  COUNT(*) as fail_count,
  COUNT(DISTINCT src_ip) as unique_ips
FROM log
WHERE
  __topic__ = 'ssh_log'
  AND action = 'Failed'
  AND __time__ > to_unix_timestamp(now()) - 3600
HAVING
  fail_count > 100
  OR unique_ips > 20
```

### Elasticsearch SIEM 规则

```yaml
# 检测可疑 PowerShell 执行
rule:
  query: |
    event.category: "process" AND
    process.name: "powershell.exe" AND
    process.command_line: (
      "* -EncodedCommand *" OR
      "* -e *" OR
      "* Invoke-*"
    )
  timeframe: 5m
  severity: medium
  tags: ["powershell", "defense_evasion"]
```

## 安全日志查询模式

### 模式1：IP 信誉关联
```
事件：SSH 登录失败 -> 提取源 IP -> 查威胁情报
结果：该 IP 被标记为 SSH 爆破，在 10 个以上的知识库中
结论：确认是恶意扫描，加入防火墙黑名单
```

### 模式2：时间聚合分析
```
事件类型：单次登录失败（正常情况）
聚合：同一 IP 在 5 分钟内失败 50 次
结果：确认是暴力破解
```

### 模式3：多源关联分析
```
Web 日志：用户访问了 /admin/login.php
WAF 日志：同一请求被标记为 SQL 注入尝试
主机日志：同时段该主机 CPU 升高
数据库日志：执行了异常查询
→ 综合判断：SQL 注入攻击成功，需要立即响应
```

## ⚠️ 日志分析常见陷阱

1. **只关注"告警"不看"正常"** — 看不到基线，就不知道什么是异常
2. **日志不全就下结论** — 缺少某个数据源的日志，攻击链就看不全
3. **日志时间不对齐** — 不同服务器的系统时间不同，关联分析时一头雾水（统一用 NTP）
4. **只看信息量不看信息质量** — 100GB 的调试日志不如 1GB 的审计日志有用
5. **日志太多不舍得删** — 但太小的保留周期又不行（等保要求 180 天）
