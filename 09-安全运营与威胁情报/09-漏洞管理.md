# 漏洞管理

> 不知道有多少漏洞、有多严重，就谈不上修复优先级。

---

## 漏洞管理生命周期

```
发现 → 评估 → 处理 → 验证 → 监控
 │      │      │      │      │
 扫描   定级   修复   复测   持续跟踪
 报告   影响   缓解   确认   趋势分析
       优先级
```

---

## 漏洞扫描

| 类型 | 说明 | 频率 |
|------|------|------|
| **网络扫描** | 扫描 IP/端口/服务，发现开放端口和已知漏洞 | 每周~每月 |
| **主机扫描** | 扫描操作系统漏洞、缺失补丁 | 每月~季度 |
| **Web 扫描** | 发现 Web 应用的 OWASP 漏洞 | 持续 / 每次上线 |
| **容器镜像扫描** | 扫描镜像中的 CVEs | CI/CD 集成 |
| **代码扫描（SAST）** | 源码级别发现安全缺陷 | 每次提交 |
| **IAST（交互式）** | 运行时结合代码分析 | 自动化测试中 |

## CVSS 评分

通用漏洞评分系统（CVSS v3.1）：

| 评分范围 | 等级 | 示例 |
|---------|------|------|
| 9.0-10.0 | 严重（Critical） | 远程代码执行无需认证 |
| 7.0-8.9 | 高危（High） | SQL 注入 |
| 4.0-6.9 | 中危（Medium） | 信息泄露 |
| 0.1-3.9 | 低危（Low） | 需要本地访问的轻微信息泄露 |

**评分不等于唯一优先级依据** — 还要考虑：
- 漏洞是否在外部可访问
- 是否有公开的利用代码
- 是否影响核心业务

## SLAs 修复时间建议

| 严重级别 | 修复 SLA | 
|---------|---------|
| 严重 | 24-48 小时 |
| 高危 | 7 天 |
| 中危 | 30 天 |
| 低危 | 90 天或下一个发布窗口 |

---

## ⚠️ 陷阱

1. **扫了不修等于白扫** — 扫描只是发现，关键是修复跟踪
2. **高危漏洞无限延期** — 需要有"达不到 SLA"的升级机制
3. **忽略历史漏洞** — 有些漏洞去年判了低危，今年有了 PoC 就变高危了
