# 安全事件案例库

> 别人的教训是最好的教材——真实的攻击案例。

---

## 数据泄露类

### 案例1：S3 桶配置错误（2020）
**涉事方：** 某大型科技公司
**原因：** AWS S3 存储桶策略设置为 public，未启用"阻止公共访问"
**泄露内容：** 上亿条用户数据（姓名、电话、地址）
**后果：** 集体诉讼，赔偿数亿美元
**教训：**
- 对象存储默认私有，需要时才开放——反过来做
- 使用 IaC 扫描工具（Checkov）检查配置
- 开启 CloudTrail + 配置告警

### 案例2：数据库公网暴露（2022）
**涉事方：** 某社交平台
**原因：** 生产数据库直接绑定公网 IP，未设白名单
**泄露内容：** 5 亿+ 用户数据
**后果：** 数据被公开售卖
**教训：**
- 数据库永不绑定公网 IP
- 使用堡垒机管理数据库
- 定期做暴露面扫描

### 案例3：第三方 SDK 供应链攻击（2023）
**涉事方：** 某数据公司
**原因：** 集成的第三方 SDK 存在数据回传后门
**泄露内容：** 用户的键盘输入、剪贴板内容
**后果：** 应用被下架，公司被调查
**教训：**
- 第三方 SDK 必须做安全审查
- 网络层面的 DLP 检测异常数据外传
- 最小权限——SDK 不应该有访问剪贴板的权限

## 勒索攻击类

### 案例4：RDP 暴露遭勒索（2021）
**涉事方：** 某能源公司
**攻击路径：** RDP 端口暴露公网 → 弱口令爆破 → 部署勒索软件
**后果：** 业务系统瘫痪 2 周，赎金 $500万
**教训：**
- RDP 永远不要暴露公网（使用 VPN 替代）
- MFA 是必须的
- 离线备份能救命

### 案例5：供应链勒索（2024）
**涉事方：** 某软件开发商
**攻击路径：** 攻击者入侵开发环境 → 在软件更新包中嵌入勒索软件 → 推送给所有客户
**后果：** 大量客户企业被加密
**教训：**
- 软件构建和发布环境需要高强度安全防护
- 代码签名验证不能少
- 软件包的哈希校验应该自动化

## 内部威胁类

### 案例6：离职员工窃取数据（2023）
**涉事方：** 某电商平台
**原因：** 离职前批量下载客户数据到个人设备
**泄露内容：** 数百万条客户交易记录
**后果：** 口碑受损，监管处罚
**教训：**
- 员工离职及时冻结账号
- 敏感数据导出需要审批
- DLP 检测异常批量下载行为

---

## 案例总结

| 教训 | 优先级 | 措施 |
|------|--------|------|
| 配置错误是最大风险 | 高 | IaC 扫描、配置基线 |
| 公网暴露要最小化 | 高 | 安全组审计、暴露面扫描 |
| 第三方依赖要审查 | 高 | SCA、SDK 审查 |
| 账号权限要管好 | 高 | IAM、离职回收 |
| 备份是最后防线 | 高 | 3-2-1 原则、定期恢复测试 |
| MFA 必须推广 | 中 | 强制开启 |
| DLP 防外泄 | 中 | 网络/终端 DLP |
