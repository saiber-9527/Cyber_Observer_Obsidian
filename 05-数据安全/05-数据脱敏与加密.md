# 数据脱敏与加密

> 让"该看的人看到该看的数据，不该看的人看到没用的数据"。

---

## 数据脱敏

### 常见方法

| 方法 | 说明 | 示例 |
|------|------|------|
| **替换** | 用假数据替换真数据 | `张三` → `李四` |
| **遮蔽** | 部分字符替换为 * | `13800138000` → `138****8000` |
| **置乱** | 随机排列数据 | 身份证号打乱再分配 |
| **哈希** | 单向哈希不可还原 | 但需要防彩虹表时加盐 |
| **泛化** | 降低精度 | `北京市朝阳区xxx` → `北京市` |
| **差分隐私** | 添加噪声 | 统计查询时加扰动 |

### 脱敏场景

| 场景 | 脱敏级别 | 说明 |
|------|---------|------|
| 开发测试 | 完全脱敏 | 不能还原出真实数据 |
| 数据分析 | 部分脱敏 | 遮蔽关键字段，保留统计特征 |
| 客服查询 | 遮蔽中间位 | 能看到尾号验证，看不到完整号码 |

## 数据加密

### 不同阶段的加密

| 阶段 | 技术 | 说明 |
|------|------|------|
| **传输中加密** | TLS 1.2+ | HTTPS，防止中间人 |
| **存储中加密** | AES-256 / SM4 | 磁盘加密、数据库加密 |
| **使用中加密** | 机密计算、同态加密 | 处理时内存也是加密的（较新） |

### 加密粒度

| 粒度 | 保护范围 | 性能影响 |
|------|---------|---------|
| **全盘加密** | 整块磁盘 | 低 |
| **数据库 TDE** | 整个数据库文件 | 中 |
| **字段级加密** | 特定字段（如身份证号） | 高（无法索引/搜索） |
| **应用层加密** | 在写入前就加密 | 最灵活 |

## 实际应用：结合使用

```
场景：一个数据分析平台需要查询用户消费记录

数据入库时：
  - 用户手机号：AES-256 加密存储
  - 用户地址：脱敏为"北京市xxx"
  - 消费金额：保留明文字段

数据分析师查询时：
  - 能看到：脱敏后的地址 + 消费金额（做分析）
  - 看不到：完整手机号、真实姓名

系统管理员：
  - 能看到加密字段的密文
  - 但没有解密密钥（存在 KMS 中）
  - 密钥需要审批才能调取
```

## ⚠️ 陷阱

1. **脱敏不等于安全** — 通过关联分析可能还原原始身份（Netflix 匿名化用户数据集被 Netflix Prize 参赛者重新识别）
2. **加密密钥管理是最薄弱的环节** — 丢了密钥=丢数据；泄了密钥=白加密
3. **不要自己发明加密算法** — 用标准算法（AES-256-GCM、RSA-2048、SM4）
