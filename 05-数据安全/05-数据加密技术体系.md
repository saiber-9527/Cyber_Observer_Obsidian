# 数据加密技术体系

> 加密不是一种技术，是一系列技术的组合，不同场景用不同的加密方式。

---

## 加密的四个层面

| 层面 | 技术 | 保护范围 | 性能影响 |
|------|------|---------|---------|
| **传输加密** | TLS 1.2+、mTLS、IPsec | 网络传输中的数据 | 低 |
| **存储加密** | 全盘加密、TDE、文件加密 | 存储在磁盘上的数据 | 中 |
| **使用加密** | 机密计算、同态加密 | 内存中正在处理的数据 | 高 |
| **应用加密** | 字段级加密、客户端加密 | 特定数据字段 | 中-高 |

## 传输加密（Data-in-Transit）

### TLS 配置参考

```nginx
# Nginx TLS 配置示例
server {
    listen 443 ssl http2;
    server_name api.example.com;
    
    # 证书
    ssl_certificate /etc/ssl/certs/server.crt;
    ssl_certificate_key /etc/ssl/private/server.key;
    
    # 只使用安全的协议
    ssl_protocols TLSv1.2 TLSv1.3;
    
    # 只使用强密码套件
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers on;
    
    # HSTS（强制 HTTPS）
    add_header Strict-Transport-Security "max-age=63072000" always;
    
    # 禁用 TLS 压缩（防 CRIME/BEAST 攻击）
    ssl_compression off;
}
```

### mTLS（双向 TLS）
```
客户端 → 服务端：客户端用服务端证书验证服务端身份
客户端 → 服务端：服务端用客户端证书验证客户端身份

适用场景：
  — 微服务间通信
  — 零信任网络
  — 需要双向认证的 API
```

## 存储加密（Data-at-Rest）

### 全盘加密 vs 数据库加密 vs 文件加密

```
┌────────────────────────────────────────┐
│        全盘加密（LUKS/BitLocker）       │
│  ├── 加密整个磁盘，OS 启动时输入密码     │
│  ├── 保护：磁盘被拔出后数据不可读        │
│  └── 缺陷：OS 运行时数据是明文的         │
├────────────────────────────────────────┤
│       数据库 TDE（透明数据加密）         │
│  ├── 加密数据库文件（.ibd/.mdf）        │
│  ├── 保护：数据库文件被拷贝也读不了      │
│  └── 缺陷：查询时的数据在内存是明文的    │
├────────────────────────────────────────┤
│       字段级加密                        │
│  ├── 在应用层加密特定字段               │
│  ├── 保护：数据库中该字段是密文          │
│  └── 缺陷：无法索引/搜索（需要特殊方案） │
└────────────────────────────────────────┘
```

### 字段级加密实现

```python
from cryptography.fernet import Fernet

# 生成密钥（存储在 KMS 中，不在代码里）
key = Fernet.generate_key()
cipher = Fernet(key)

# 加密（存入数据库前）
encrypted_phone = cipher.encrypt(b"13800138000")
# 存储 encrypted_phone 到数据库

# 解密（从数据库读出后）
decrypted_phone = cipher.decrypt(encrypted_phone)
# 在应用中展示解密后的数据
```

## 使用中加密（Data-in-Use）

最前沿的加密技术，解决"内存中的数据也是加密的"。

### 机密计算（Confidential Computing）
- CPU 提供 **TEE（可信执行环境）**
- 数据在 CPU 内部解密，内存中始终保持加密
- 即使是云管理员也看不到你的数据
- 代表技术：Intel SGX、AMD SEV、ARM TrustZone

### 同态加密（Homomorphic Encryption）
- 直接在密文上进行计算，结果解密后等于对明文计算的结果
- 目前性能开销很大（加密状态的计算比明文慢 1000+ 倍）
- 适用场景：医疗数据统计分析、金融数据联合建模

## 加密选型指南

```
需要保护传输中的数据 → TLS 1.2+
服务器硬盘可能被拔 → 全盘加密（LUKS/BitLocker）
数据库文件可能被拷贝 → TDE（透明数据加密）
数据库管理员不应看到敏感数据 → 字段级加密
需要加密同时要搜索 → 可搜索加密（Searchable Encryption）
云厂商不应看到客户数据 → 机密计算（TEE）
需要多机构联合计算 → 联邦学习 / 安全多方计算
```

## ⚠️ 加密常见错误

1. **加密了 key 没管好** — 密钥泄露比不加密更糟（虚假的安全感）
2. **使用了不安全的算法** — DES/MD5/SHA-1 还在用
3. **自己造加密算法** — 99% 的情况，标准算法就够了
4. **加密了但没防侧信道** — 常量时间算法的重要性
5. **TLS 配置太低** — 仍然允许 TLS 1.0 或弱密码套件
